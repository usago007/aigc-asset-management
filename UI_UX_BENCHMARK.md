# FatMug UI/UX 验收基线

## 目标

FatMug 不以“像某个 AI Playground”为验收标准，而以可度量的企业级内容生产闭环为标准：创作、项目、镜头、资产、审核、交付和治理必须处于同一工作上下文。

## 核心基线

| 维度 | 验收要求 | 当前证据 |
| --- | --- | --- |
| 创作效率 | 图片和视频任务可并行排队；模型、队列、历史结果和业务绑定同屏可见 | `CreationTelemetry`、图片/视频工作台 |
| 业务闭环 | 生成结果可追溯到项目、镜头、模型、任务和资产 | 项目详情、镜头详情、图片/视频详情、资产库 |
| 数据解释 | 指标必须包含颗粒度、分母或业务判断 | 数据中心四个分析视图 |
| 操作安全 | 删除和解绑必须使用可取消的上下文确认，不使用浏览器原生弹窗 | `ConfirmProvider` 与管理页操作链 |
| 响应式 | 768px 下无页面横向溢出；高密度表格首列和操作列可持续访问 | `projects-mobile-light.png` 与浏览器验收 |
| 主题一致性 | 亮色、暗色共用相同信息层级、对比度和交互状态 | 数据中心暗色基线与主题验收 |
| 无障碍 | 主内容跳转、可见焦点、语义导航、按钮名称、图片替代文本完整 | 全局壳层与 `verify:ui` |
| 可验证性 | 主要页面契约、真实浏览器路由和视觉快照均可重复生成 | `verify:ui`、`verify:routes`、`baseline:ui` |

## 差异化标准

- 创作不是孤立 Prompt 会话：每次生成可绑定项目与镜头，并进入资产和审核链路。
- 数据不是装饰性图表：每个指标说明计算口径，并提供可行动的下钻入口。
- 历史结果不是图片墙：支持重新编辑、再次生成、查看详情和技术追溯。
- 企业治理不是后置页面：成员、角色、环境配置、日志和删除安全属于统一产品体验。

## 必过命令

```bash
npm run verify:ui
npm run build
npm run verify:routes
npm run baseline:ui
```

任何新增一级页面必须使用统一标题区并说明页面用途；任何新增路由必须加入真实浏览器路由验收；任何视觉系统变更必须更新视觉基线。
